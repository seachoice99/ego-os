# ADR-0012: Multi-executor task dispatch is sequential, tiered by task complexity, never parallel-checkout

## Status

Accepted by Owner on 2026-07-12, as part of a new Multi-Executor Dispatch initiative building on Epic 2 (Runner Integration, ADR-0009) and Epic 3 (Usage Optimization, ADR-0010).

## Context

The Owner holds both a Claude subscription and a Codex (OpenAI) subscription, and wants both to participate in developing this repository, plus free-tier OpenRouter models for low-stakes work — ranked explicitly by priority: (1) minimize token/usage-limit consumption above all else, (2) split responsibility between Claude and Codex with no conflicts, (3) tasks are picked up and run without the Owner's active involvement, within whatever limits exist, (4) simple tasks where model quality doesn't matter go to free OpenRouter models.

ADR-0009/`architecture/015_RUNNER_OPERATIONS_INTEGRATION.md` already added the `executor: claude|codex|auto` schema placeholder and a fail-closed rule for it, but explicitly deferred building Codex ("no Codex implementation exists yet... this epic measures and prepares, it does not build Codex" — ADR-0010). `RCI-01`/`RCI-02` (the tasks that would add and enforce this schema) are still `status: "blocked"`, and grep confirms zero references to `executor` anywhere in `automation/`'s runtime code today.

Investigation before this ADR (two Explore passes over `automation/claude_task_runner.js`, `automation/session_manager.js`, `automation/control_server.js`, `automation/windows_agent.js`, `ego_os/model_provider.py`, `ego_os/agent_routes.py`) established the concrete facts this decision rests on:

- `runClaude()` (`claude_task_runner.js`) is binary-agnostic (`cp.spawn(BINARY, args)`); only `claudeInvocationArgs()` (`session_manager.js:268-272`) and the module-level `CLAUDE` binary constant are Claude-CLI-specific.
- `control_server.js` already maintains a **registry of agents keyed by `agent_id`, each carrying its own `executor` field** (`summarizeAgents()`) — used today only for display, never to filter which task an agent may claim.
- `preflight()` requires a clean working tree and `local HEAD == origin/main` before any task starts, but this is a per-task-start check, not a cross-process mutex; `save()` on the task YAML is a plain, unlocked `fs.writeFileSync`. Two independent CLI processes (one Claude, one Codex) editing the same checkout at the same time is a real, demonstrable conflict source — not a hypothetical one.
- The Codex CLI is not installed on the Owner's machine (confirmed live: `command -v codex` and `Get-Command codex` both fail). Its actual non-interactive invocation flags, output format, and rate-limit/auth-failure signals are unverified. Writing Claude-style fatal-pattern/rate-limit detection for Codex without ever having run it would violate this project's own "no estimates presented as measurements" discipline (Epic 3/UOP).
- `ego_os/model_provider.py` already has a working OpenRouter client (`_get_client()`, `complete(capability, prompt, ...)`), but it serves the in-app product features, not the autonomous queue runner — the queue needs its own lightweight, non-agentic path.

## Decision

1. **Sequential dispatch, never two agentic CLI processes writing to the same checkout concurrently.** One dispatcher loop (the existing `claude_task_runner.js`/`windows_agent.js` claim-execute cycle) picks the next eligible task and spawns exactly one child process for it, choosing the binary/flags by that task's `executor`. Running Claude and Codex "at the same time" in this system means alternating turns on the same shared repository, never two simultaneous working-tree writers — this is what makes "no conflicts" (priority 2) true by construction rather than by locking discipline that could fail under a race.
2. **Three complexity tiers, not a "which vendor is smarter" debate.** Tier 0 (mechanical/low-ambiguity: reading logs, status checks, doc-only edits) routes to `executor: openrouter_free` — a direct, non-agentic OpenRouter API call, never touching `ego_os/`, `automation/`, or tests (Owner-set scope boundary), reusing the capability-routing pattern already described in `models/MODEL_SELECTION_POLICY.md` and `architecture/016`. Tier 1 (ordinary feature work with clear acceptance criteria) routes to `executor: claude` or `executor: codex`, chosen by whichever currently has usage-limit headroom. Tier 2 (architecture, complex or risky changes) additionally gets an independent review stage from the *other* executor, via the same handoff-file mechanism TOKEN-EFFICIENCY-001 already uses for stage-to-stage continuation — this satisfies ADR-0010 principle 5 (independent review for complex changes) at no extra dollar cost, since the reviewing subscription is already paid for.
3. **Executor choice is explicit and disclosed, never a silent auto-switch**, per ADR-0010 principle 4. A task's author sets `executor`/`model_tier` explicitly; `executor: "auto"` resolves through a pure, unit-tested, documented function (`chooseExecutor`) — never an undisclosed heuristic. The one automatic behavior this ADR authorizes is: if the chosen executor is currently `waiting_for_limit`/`waiting_for_auth`, fall back to the other real executor automatically — a concrete, logged rule, not silent guessing. This is what makes priority 3 ("without the Owner's active involvement, within existing limits") true without violating principle 4.
4. **Codex integration is not written from memory.** Before any Codex-invocation code exists, a read-only recon task (`MED-01`) installs/probes the real Codex CLI and records its actual non-interactive flags and failure signals. `RCI-02`'s fail-closed executor validation rejects both `codex` and `openrouter_free` (in addition to any unrecognized value) until each has a real, tested implementation — mirroring the existing RUNNER-CONTROL-UI fail-closed precedent (an unimplemented capability must error loudly, never silently no-op or silently run as Claude).
5. **`openrouter_free` gets a hard scope boundary, set by the Owner**: read/analysis and documentation-only edits (`docs/`, `product_bible/`, `CHANGELOG.md`, log/status reads). It never gets `allowed_paths` inside `ego_os/`, `automation/`, or any test file — a weaker, unreviewed-by-a-second-pass model should never be the sole author of runtime code or tests in this system.

## Consequences

- No new concurrency-control code (distributed lock, CAS, worktree isolation) is needed to satisfy "no conflicts" — the sequential-dispatch decision makes conflicts structurally impossible instead of merely improbable. This trades away same-instant parallel throughput, which is not something priorities 1-4 asked for.
- The existing agent registry (`control_server.js`'s `agents` map, already keyed by `agent_id` with an `executor` field) becomes load-bearing rather than decorative once claim filtering is implemented (`MED-06`) — no new registry design was needed.
- Real Codex-path code (`MED-02` onward) is blocked on `MED-01`'s findings by design; this ADR explicitly accepts that Codex participation does not exist yet as of this decision, matching ADR-0010's own precedent of not building Codex ahead of verified facts.
- `openrouter_free`'s restricted scope means priority 4 ("free models for simple tasks") is satisfied only for a narrower task set than "any simple task" might suggest — mechanical code edits still go through Claude or Codex. This is a deliberate safety trade the Owner made explicitly (asked directly, chose the restricted-scope option over full repo access for free-tier models).
- `architecture/017_MULTI_EXECUTOR_DISPATCH.md` carries the full schema and routing contract this decision authorizes.
